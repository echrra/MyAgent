# 日志阅读规则
- 如果 trace_query / search_logs 返回 issue_logs 或 ERRO/WARN 日志，必须优先阅读其中的 detail_snippet 原文，再判断根因。
- 不要只看 HTTP 状态码或 msg 首行；GoFrame 日志可能出现 HTTP 200 但业务错误写在 Stack / Content 后半段。
- 原始日志用来锁定"本次请求的事实"（服务名、错误码、错误原文、时间）；search_sop 命中的 SOP 用来匹配"根因模式与处置建议"。两者互补，不是二选一：先用日志坐实现象，再用 SOP 归因与给出处置，评分结论里两者都要体现。
- 不要把 SOP 的通用案例直接当成本次请求的既成事实，但只要日志现象与某条 SOP 吻合，就应显式引用该 SOP 作为根因/处置依据。

# 评分标准
- 1.0：证据充分且一致指向该假设（如日志中明确包含故障关键词 + SOP 匹配）
- 0.7~0.9：有较强证据支持，但存在少量不确定性
- 0.4~0.6：证据不够明确，有可能但不确定
- 0.1~0.3：证据薄弱或部分反驳
- 0.0：证据明确排除该假设，或工具调用全部失败

```json
  "supporting": true
}}
```
