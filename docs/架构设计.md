# OpsAgent 架构设计

> 本文整合项目定位、技术选型、系统链路、记忆系统与数据/评测五部分，是理解代码库的主文档。
> 描述以实际代码为准；具体实现见 `opsagent/` 各模块。

---

## 一、项目定位与边界

OpsAgent 是一个面向微服务架构的智能运维诊断 Agent。输入一段故障描述或告警，
系统自动生成故障假设、并行调度工具验证、综合证据给出带引用的根因结论。

**核心场景**

| 场景 | 说明 |
|---|---|
| 跨服务故障排查 | 根据告警/描述，串联日志、Trace、指标、变更多源数据，给出根因猜想 |
| 日志根因定位 | 从海量日志中圈定异常时间窗、相关服务、关键错误链 |
| SOP 知识检索 | 基于历史复盘库与运维文档，检索相似故障与处置预案 |
| 运维问答 | 服务拓扑、配置项、链路依赖等自然语言查询 |

**能力边界（刻意收窄，避免失焦）**

- ✅ 做：多源数据查询与聚合、多步推理（理解→规划→工具→反思→应答）、引用溯源、跨会话记忆、评测→复盘→回归闭环。
- ❌ 不做：自动化执行（不重启服务/不改配置）、模型训练/微调、日志采集与告警平台本身（不重造 ELK/Prometheus）、认证权限多租户、推理优化。

**数据来源**：检索底座为公开数据集，评测数据为 LLM 合成，故障形态提炼自真实生产经验（详见第五节）。

## 二、技术选型

每一项选型的标准是：社区成熟、生态标准、能一句话讲清"为什么是它"。

| 维度 | 选型 | 理由 |
|---|---|---|
| 语言 | Python 3.11+ | Agent 生态最成熟 |
| 编排框架 | **LangGraph** | 图编排 + State + Checkpoint 是标准范式；节点级流式天然适配工具进度推送；Send API 支持并行 fan-out |
| Web 框架 | FastAPI | 异步 + Pydantic + SSE 原生友好 |
| 模型 | DeepSeek / Qwen 系列 | 通过 OpenAI 兼容层统一调用，便于主备 fallback |
| 模型网关 | LiteLLM | 多供应商切换、统一计费与超时重试 |
| 向量库 | **pgvector** | 元数据（服务名/时间窗/等级）与向量同库 join，一站 SQL 搞定，不维护两套数据一致性 |
| 稀疏检索 | BM25S | 轻量，不引入 ES |
| 融合 | RRF（k=60） | 公式简单、可解释，比加权求和鲁棒 |
| 精排 | BGE-Reranker-v2-m3 | 开源工业标杆，Top-50 → Top-5 |
| 工具协议 | MCP × 2 + Function Calling × N | 双轨：对外可复用能力走 MCP，内部工具走 FC |
| 可观测 | Langfuse | Trace/Span/Score 全链路，Badcase 反向定位 |
| 评测 | 自建评测集 + Ragas 双指标 | 主线自建；Ragas 仅 faithfulness / context_recall 做趋势观测 |
| 前端 | Chainlit | 流式 + 工具调用可视化开箱即用 |
| 流式协议 | SSE | Agent 是单向推送场景，SSE 够用且工程更简单，网关/CDN 友好 |

**关键选型说明**

- **编排框架**：LangChain 的 Chain 抽象在多分支/多 Agent 场景表达力不足；LangGraph 用图显式描述状态流转，可观测性与可恢复性（Checkpoint）更好。
- **检索四件套**：稀疏（BM25S）+ 稠密（pgvector + bge embedding）+ 融合（RRF）+ 精排（Cross-Encoder）。向量召回偏语义相似，但"相似 ≠ 相关"，精排用 Cross-Encoder 显式建模 query-doc 交互。
- **为什么不全 MCP 化**：MCP 的价值在跨进程/跨语言/协议统一；对内部 Python 工具，FC 更简单，没必要为协议而协议。
- **明确不引入**：Milvus/Weaviate（pgvector 够用）、Elasticsearch（BM25S 够用）、Kafka（不做异步管道）、SFT/vLLM（应用层项目）。

## 三、系统架构与链路

### 3.1 整体分层

```
┌───────────────────────────────────────────────┐
│              Chainlit / Web UI                  │  用户输入 / 流式 Token / 工具进度
└───────────────────────┬─────────────────────────┘
                        │ SSE
┌───────────────────────▼─────────────────────────┐
│                  FastAPI 入口                    │  会话管理 / SSE 推送
└───────────────────────┬─────────────────────────┘
                        │
┌───────────────────────▼─────────────────────────┐
│                LangGraph 编排层                  │  两套图，配置切换
└───────────────────────┬─────────────────────────┘
        ┌───────────────┼───────────────┬───────────────┐
        ▼               ▼               ▼               ▼
   ┌─────────┐    ┌──────────┐    ┌──────────┐    ┌────────┐
   │ 记忆层  │    │ 工具层   │    │ 检索层   │    │ 模型层 │
   │ Memory  │    │ MCP + FC │    │ 混合 RAG │    │ LiteLLM│
   └────┬────┘    └────┬─────┘    └────┬─────┘    └───┬────┘
        └──────────────┴───────────────┴──────────────┘
                        │
   ┌────────────────────▼────────────────────────────┐
   │   PostgreSQL + pgvector │ 数据集 │ Langfuse Trace │
   └──────────────────────────────────────────────────┘
```

### 3.2 两套编排图（`opsagent/core/graph/builder.py`）

代码内置两套图，由 `graph_version` 配置切换（默认 v2）：

**v1 — 单 Agent ReAct**（`_build_graph_v1`）

```
load_memory → plan_node → [tool_exec → reflect] ↺ → answer_node → persist_memory
```

Plan / Tool / Reflect / Answer 四节点分离，各自可单测、可观测；Reflect 判断证据是否充分，
循环上限 5 次后强制 Answer 兜底。

**v2 — Multi-Agent 假设驱动并行诊断**（`_build_graph_v2`，当前默认）

```
load_memory → coordinator ──Send fan-out──▶ worker × N（并行） ──barrier──▶ synthesizer → persist_memory
```

- **Coordinator**：理解 query，产出多个故障假设（每个假设一个诊断方向）。
- **Worker**：每个假设一个 Worker，用 LangGraph Send API 并行 fan-out，各自独立调工具验证本假设。
- **Synthesizer**：barrier 汇聚所有 Worker 结果，比对证据，生成带引用的最终结论。

> v1→v2 的升级动机与数据对比见《开发历程.md》。两套图共享记忆层与工具层。

### 3.3 单次查询时序（v2）

```
用户 ─POST /chat─▶ FastAPI ─invoke─▶ LangGraph
                                      │
                                load_memory：装配四层记忆上下文
                                      │
                                coordinator：生成 N 个故障假设 ──▶ SSE: thinking
                                      │  Send fan-out
                        ┌─────────────┼─────────────┐
                     worker_1      worker_2   ...  worker_N（并行调工具验证）──▶ SSE: tool_*
                        └─────────────┼─────────────┘  barrier
                                synthesizer：比对证据 + 引用溯源 ──▶ SSE: token（流式）
                                      │
                                persist_memory：落库 + 异步触发画像更新/压缩
                                      │
                                    ──▶ SSE: done
```

全链路每个节点一个 Langfuse span，Badcase 复盘时按 session 拉 trace 逐 span 回放。

### 3.4 State Schema 要点（`opsagent/core/graph/state.py`）

`AgentState` 贯穿全图，关键字段分组：

- **会话标识**：`session_id / user_id / trace_id`
- **上下文装配**：`system_prompt`（L1）、`profile_context`（L2）、`episodic_messages`（L3）、`working_memory`（L4，`Annotated[..., add]` 本轮累加）
- **推理状态**：`plan / next_action / iteration`（v1）；`hypotheses / worker_results`（v2）
- **检索与输出**：`retrieved_docs`、`final_answer`、`citations`（每条含 source_id + snippet）
- **控制**：`error`、`should_compact`

### 3.5 工具层（`opsagent/core/tools/`）

- **Function Calling 工具**：`search_logs`、trace 聚合、`create_ticket`（确定性 ticket_id）、变更查询等——见 `mock.py` / `real.py`。
- **MCP Server × 2**：`kb_server.py`（知识库检索，FastMCP 包装检索流水线）、`metrics_server.py`（监控指标查询）。
- **Tool 抽象层**（`base.py`）统一处理：失败 Retry 1 次、超时控制（默认 10s）、Pydantic 参数校验、输出截断（避免大块文本污染上下文）、Trace 自动打点。工具最终失败时 `success=False`，由 reflect/plan 改道，不直接崩。

## 四、四层记忆系统

业界常见的"长短期"二分过粗。本项目按**生命周期、写入时机、检索方式、压缩策略**四个维度拆成四层，分开管理。实现见 `opsagent/core/memory/`。

| 层 | 名称 | 内容 | 生命周期 | 存储 |
|---|---|---|---|---|
| L1 | **System** | 角色定义、能力描述、安全约束 | 不变 | 代码/配置 |
| L2 | **Profile** | 用户画像、长期偏好、历史故障模式 | 跨会话持久 | pgvector + Postgres |
| L3 | **Episodic** | 对话历史（含压缩摘要） | 单会话 | Postgres |
| L4 | **Working** | 当前轮工具调用结果、中间推理 | 单轮 | LangGraph State（内存） |

### 4.1 L1 System（`system.py`）

三段式动态装配：角色 + 能力（动态拼接当前可用工具）+ 安全约束（"不编造日志内容，结论必须有引用"）。
按用户角色调整语气：SRE 用专业术语、新人避行话、Manager 偏结论。

### 4.2 L2 Profile（`profile.py`）

- 结构化字段（角色、关注服务、沟通偏好）按 `user_id` 直接 SQL 查。
- 历史故障模式向量化存储，按当前 query 语义检索命中后追加到上下文——这是选 pgvector 的直接理由（同库 join）。
- 写入：会话结束时异步触发 Profile Updater 从对话抽取偏好，不阻塞主链路。
- **冲突处理**：不覆盖，多版本 + 时间戳，最新优先；在上下文里让 LLM 看到冲突（如"角色历史：SRE(today), backend(last week)"）而非后端硬选。

### 4.3 L3 Episodic + 压缩（`episodic.py` / `compactor.py`）

装配顺序：`[System] [Profile] [Summary_old] [Recent N rounds] [Working]`。

**压缩触发**（任一满足）：token 占比 ≥ 0.7 / 累计 ≥ 8 轮 / 手动 `/compact`。阈值不写死，从模型 `max_context` 按比例算，配置驱动。

**压缩策略**：
- 只压 LLM 输出与工具长输出，**不压用户消息**（用户原话是不可丢失的事实输入）。
- 保留 System + Profile + 最近 2-3 轮完整对话。
- 摘要为结构化 JSON（`topic / key_facts / tools_used / decisions / unresolved / rounds_compressed`），可字段级检索回填，比纯文本摘要可控。
- 多次压缩后**递归压缩**：一级摘要累计达阈值合并为 level2，旧的标 superseded 不删。
- 压缩是 fire-and-forget 异步任务，失败时原始消息不丢，走截断兜底。

### 4.4 L4 Working（`working.py`）

本轮 Plan-Tool-Reflect / Worker 循环内累积的工具调用记录（args/result/success/latency）与中间推理。写入频繁但几乎不持久化。与 Trace 的关系：同一份数据两个出口——Working 给"当前 LLM"看（推理依据），Trace 给"工程师"看（事后复盘）。

### 4.5 Memory 与 RAG 的区别

| 维度 | Memory | RAG |
|---|---|---|
| 数据归属 | 用户/会话私有 | 公共知识库 |
| 写入 | Agent 自动写（每会话/每轮） | 离线 ingest（低频） |
| 检索粒度 | 个性化（按 user_id） | 全局（按 query） |
| 生命周期 | 有过期/压缩/合并 | 通常静态 |

两者共用同一份 pgvector 实例，通过 schema/table 隔离。

## 五、数据策略与评测

### 5.1 三层数据架构

```
层 1 公开数据集（检索底座）  ── LogHub 等公开日志子集，给检索"装真血"
        ▼
层 2 业务样本抽象（风格锚点）── 人工通读生产样本，抽 schema 与故障短语，占位符化后以 yaml 入库
        ▼                     （原始日志绝不入仓；yaml 抽象产物才是合成器的输入）
层 3 LLM 两阶段合成（规模化） ── Generator 仿写 + Critic 判分，合格入库；叠加 F1-F13 故障信号
```

| 层 | 量级 | 用途 | 入评测集 |
|---|---|---|---|
| 层1 公开数据 | 大规模 | 检索语料底座 | ❌ |
| 层2 业务抽象 | 数十短语/服务 | schema + few-shot 范例 | ❌ |
| 层3 LLM 合成 | 10w 行 + 多类故障 | 评测 ground truth + 工具查询对象 | ✅ |

**合成原则**：结构对齐生产（日志 OpenTelemetry 风格、Trace W3C Trace Context、指标 Prometheus 命名）；形态对齐真实故障经验；业务壳虚构；`--seed` 可重放。故障模式（级联超时、Kafka 异常、慢 SQL、缓存击穿、DNS 失效、Token 解码错误等）见 `data/synthesizer/fault_patterns/` 与 `data/docs/`。

### 5.2 评测体系

四层评测：L4 端到端任务正确率（核心）/ L3 检索质量 / L2 Ragas 双指标 / L1 单测。

**端到端评分公式**（`eval/metrics/`）：

```
score = 0.4 * tool_sequence_score + 0.3 * citation_score + 0.3 * conclusion_score − forbidden_penalty
```

- `tool_sequence_score`：期望工具集合的 F1（顺序仅作 tiebreaker，避免惩罚合理的多调工具）。
- `citation_score`：梯度评分——精确命中 1.0 / 同域 0.5 / 缺失 0。
- `conclusion_score`：命中关键词数 / 期望最少数，封顶 1.0。
- `forbidden_penalty`：命中禁忌词扣分（防幻觉、防"已自动重启"这类越界结论）。

**评测集**（`eval/dataset/cases/`）：从故障复盘文档逆向推导 query，同一故障写多角色版本（SRE 专业 / 业务化 / 新人），按 easy/medium/hard 分级。

**Badcase 闭环**：评测跑完拉总分低的 case → 打开 Langfuse trace 逐 span 定位失败节点（Plan 错/工具错/检索 miss/反思未止）→ 改 prompt 或约束 → 加回评测集做永久回归。

## 七、数据资产说明

数据是本项目的核心资产之一，分为**知识库语料**、**日志数据**、**评测集**、**合成脚本**四大块。下面逐一说明每类文件的用途。

### 7.1 知识库语料 `data/docs/`（RAG 检索底座，全部手写入库）

这是 Agent 做 SOP 检索、根因关联时实际检索的知识库，建索引后进入 pgvector + BM25。

| 目录 | 内容 | 数量 | 说明 |
|---|---|---|---|
| `services/` | 服务说明文档 | 8 | 每个服务（`edgectl-gateway`/`edgectl-admin`/`edgectl-ugc` 等）的职责、依赖、常见故障位置 |
| `sops/` | 标准排查指南（SOP） | 13 | 按故障模式组织（`sop-f1` 级联超时 … `sop-f13` 401 扩散），含适用场景、排查步骤、常见根因、处置预案 |
| `postmortems/` | 历史故障复盘 | 12 | 真实故障场景的复盘案例（现象 → 影响 → 排查 → 根因 → 修复），是评测 query 的逆向来源 |
| `runbooks/` | 中间件运维手册 | 4 | Kafka / MySQL / Redis / Prometheus 的运维要点 |
| `symptoms.yaml` | 口语症状旁路表 | 1 | "口语症状 ↔ 技术正文"映射，用于 chunk 整形，缓解 reranker 对不对称 pair 打分偏低的问题 |
| `README.md` | 语料索引 | 1 | 文档间交叉引用关系说明 |

> SOP / postmortem 之间有交叉引用（SOP 指向相关复盘、复盘标注关联 SOP 与 Fx 故障模式），构成知识闭环。

### 7.2 日志数据 `data/logs/`（三层数据架构，对应第五节）

| 目录 | 层级 | 内容 | 是否入库 |
|---|---|---|---|
| `real/{OpenStack,Spark,Zookeeper}/` | **层1 公开数据集** | LogHub（CUHK 维护，AIOps 学界标准）的真实生产日志子集，作检索底座"装真血"。下载方式见 `real/README.md` | gitignore（按需下载） |
| `seeds/*.yaml` | **层2 业务抽象** | 人工通读真实业务样本后抽取的字段 schema 与故障短语（占位符化）。**原始日志绝不入仓**，只入 yaml 抽象产物，作为合成器 few-shot 范例 | 仅 yaml |
| `skeletons/*.jsonl` | 合成中间态 | 各故障模式的日志骨架（`fault_F*_skeleton.jsonl`），合成流水线的中间产物 | ✅ |
| `synth/*.jsonl` | **层3 LLM 合成** | 最终供 Agent 查询、供评测用的合成日志。命名 `fault_F{n}_claude.jsonl`；带 `_real_` 的（如 `fault_F11_real_requeue_storm`）是真实环境样本经脱敏+仿写后的产物；带 `_seed{n}` 的是同故障不同随机种子的可复现副本 | ✅ |

> **真实生产/uat 日志**（原 `real/prod/`、`real/uat/`）含未脱敏用户 PII，**已从交付库移除**。如需重新接入，见《使用说明.md》「真实环境日志接入」——通过 `scripts/fetch_uat_tls_logs.py` 从火山引擎 TLS 拉取，再经 `scripts/tls_to_search_logs.py` 脱敏转换后落到 `synth/`。

### 7.3 其它数据目录

| 目录 | 内容 |
|---|---|
| `data/traces/`、`data/metrics/` | 预留的 Trace / 指标数据目录（当前以 `.gitkeep` 占位，运行时产物 gitignore） |
| `data/synthesizer/` | 数据合成脚本：`run_all.py`（一键合成）、`download_loghub.py`（下载层1公开数据）、`critic_only.py`（Critic 判分），`fault_patterns/`（F1–F13 各故障模式的信号注入器） |

### 7.4 评测集 `eval/dataset/cases/`（评测 ground truth）

61 个 YAML 用例（`E001.yaml …`），每条含 query、user_profile、期望工具序列、期望引用、期望结论关键词、禁忌词。从 `postmortems/` 复盘文档逆向推导，同一故障写多角色/多难度版本。评分逻辑见 `eval/metrics/`，执行器 `eval/runner.py`，报告输出到 `eval/reports/`（运行产物，随跑随生成）。
