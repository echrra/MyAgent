# OpsAgent 开发历程

> 记录项目从单 Agent 到多 Agent 的几个关键演进节点。每个节点保留了"为什么改、改了什么、数据如何"的工程判断，作为设计决策的追溯依据。

---

## 阶段一：单 Agent ReAct 打通（v1）

先把最短闭环跑通：`load_memory → plan → tool → reflect → answer`，SSE 流式 + Chainlit 前端。
Plan / Tool / Reflect / Answer 四节点分离，每步可单测、可观测；循环上限 5 次强制收敛，防死循环。

这一阶段目标是"跑通"而非"跑好"，为后续检索、记忆、评测各模块提供承载骨架。

---

## 阶段二：混合检索 RAG（BM25 + 向量 + RRF + Rerank）

把 SOP / 复盘文档建成可检索知识库，四段检索全部落地。此阶段有一个反直觉的实测发现：

| 评测集 | Recall@5 | MRR(rerank off→on) | 结论 |
|---|---|---|---|
| v1：query 含精确术语（错误码、服务名） | 100% | 1.000→1.000 | rerank 零差异 |
| v2：抹掉精确词、只留口语症状 + 语义干扰 | 100% | 0.920→0.870 | rerank 未拉升、略降 |

**两个工程结论**：

1. **query 含精确术语时，BM25 召回阶段就把 gold 顶到 rank1，rerank 没有发挥空间**——漂亮的 100% 其实是"考题泄题"，做难评测集才有意义。
2. **暴露了 pipeline 漏做按文档去重**：一篇文档切多 chunk，召回时多个 chunk 挤占 top5（实测某 query 的 top5 全是同一篇），且导致 NDCG 误算 >1。修复：精排后按 `doc_id` 去重，每篇只留最高分 chunk。

诚实结论：小语料 + 症状互斥场景下 rerank 跑不出正向增量是合理的——召回已近满分无拉升空间，cross-encoder 在语义暧昧 query 上甚至引入噪声。**rerank 的价值随语料规模、候选噪声、query 歧义度增大而显现**，这是"何时该上 rerank"的工程判断。

---

## 阶段三：四层记忆系统

不使用 LangGraph 自带 memory 黑盒，按生命周期自研四层（System / Profile / Episodic / Working），
含压缩触发器与递归压缩（详见《架构设计.md》第四节）。

验收以真实 PG + 真实 LLM 端到端为准：压缩触发器（轮数 ≥ 8 / token 占比 ≥ 0.7）实测命中并生成结构化摘要 + 递归合并；跨会话按 `user_id` 拉回多版本画像；同 session 第二次请求从真实 PG 读回上一轮对话拼进上下文。

已知边界（诚实标注）：fire-and-forget 用 `create_task`，正式环境应换后台任务队列；意图门控召回、token 级精确裁剪、profile TTL 未做。

---

## 阶段四：评测体系与评分公式迭代

评分公式经过多轮迭代，核心是"评估诊断方向而非工具名"：

- **首版问题**：F1 公式惩罚多调工具（Agent 调 4 个但 expected 只写 2 个 → F1 虚低）；citation 二值评分导致同一 case 重跑分数抖动；forbidden 一刀切（命中 1 词扣 0.5，一个误判毁掉整条）。
- **修复方向**：工具分改为集合 F1（顺序仅作 tiebreaker，不惩罚合理的多调）；引用改**梯度评分**（精确 1.0 / 同域 0.5 / 缺失 0）；禁忌词惩罚分级。
- **关键认知**：首批低分中相当一部分是**评测标准不合理**在惩罚正确行为，而非 Agent 能力不足——评测集本身也需要被评测和迭代。

全链路接入 Langfuse（每节点一 span），Badcase 可按 session 拉 trace 逐 span 回放定位。

## 阶段五：单 Agent → Multi-Agent 假设驱动并行（v2 图）

**升级动机**（有真实数据支撑，不是为多而多）：单 Agent 在 Plan 节点要同时权衡"查日志/指标/trace/变更"四件事，提示词过长导致选择质量下降；顺序调用导致一次会话 5+ 轮工具调用，token 涨得快且中途易走偏。

**改造**：拆成 Coordinator（生成多故障假设）→ Worker（每假设一个，Send API 并行验证）→ Synthesizer（barrier 汇聚、比对证据、引用溯源）。两套图用 `graph_version` 开关保留，可做 A/B 对比。

**效果与新问题**（4 次全量评测）：

| 指标 | 单 Agent(v2 评分) | Multi-Agent | 变化 |
|---|---|---|---|
| 平均分 | 0.765 | 0.784 | +2.5% |
| 工具覆盖率 | 0.516 | 0.988 | +91% |
| 引用率 | 0.762 | 0.673 | **−11.7%** |

Multi-Agent 解决了工具覆盖问题，但引入了**多跳信息传递的质量衰减**：Coordinator 的假设描述 → Worker 的检索 query → Synthesizer 的引用，链路越长噪声越大。引用率还出现明显双峰波动（Run 间极差 16%）。

**根因定位**：Worker 的 `search_sop` 直接用 Coordinator 生成的自然语言假设描述做 query，**绕过了单 Agent 路径里的 Query Rewriting 环节**；而引用是 0/1 二值放大器，小样本内极差被放大。

---

## 阶段六：cite 双峰根因定位

双峰波动的根因经过多轮控制变量实验，结论几经反转，最终定位为：

- **不是**假设非确定性、**不是**检索质量本身、**微调也治不了**；
- 实锤为**托管 LLM serving 层的数值抖动**（换官方端点复现、固定 seed 压不住、temperature 已为 0）。

**对策**：走工程免疫而非模型侧——检索锚定原始 user query（不用不稳定的假设描述）+ Synthesizer temperature→0。

> 这一节的价值在于方法论：观察异常 → 控制变量实验 → 定位根因 → 针对性修复 → 多次验证，而非拍脑袋改公式。中间被推翻的结论也是过程的一部分。

---

## 阶段七：真实环境日志接入与 PII 脱敏

将真实生产日志（linkcraft，见《架构设计.md》命名说明）接入 Agent 验证，踩到一个隐蔽的坑：

**LLM provider 对结构化 PII 会静默拒答**——真实日志每条 msg 带完整 JWT，payload base64 解码后是明文 userId/deptId/mobile 组合。provider 内容安全策略识别到结构化 PII 后：请求成功、耗时正常、无 429、无 error code，但 **stream 返回 0 token**，观测层看着像"Agent 判定证据不足主动放弃"。

**这个失败模式评测集验证不出来**——评测数据是自己合成的，不会踩到这个雷；只有接真实数据才暴露。

**修复与设计要点**（`opsagent/core/utils/pii_sanitizer.py`）：

1. 加一层强制正则脱敏层，真实结果落回 Agent 前必过 sanitizer。
2. **脱敏 ≠ 无脑遮盖**：保留 JWT header 的 `kid`/`alg`（对诊断有价值）——正是靠保留 `kid` 才能让 Agent 关联到密钥轮换类 postmortem。合规和诊断在这一层可以同时兼顾。
3. **顺序 bug**：预览截短必须放在脱敏之后，否则 200 字截点会把 JWT 切两半、正则匹配不上（用 `grep -c "eyJ"` 反查残留验证）。
4. **service 改名兜底**：真实服务名与评测集不同，用 adapter 层改名表接入（比改 Agent 代码干净），保留 `service_real` 字段作追溯。
