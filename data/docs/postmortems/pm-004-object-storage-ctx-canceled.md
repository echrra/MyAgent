# 故障复盘 PM-004：大文件上传被网关超时掐断

## 基本信息

| 项目 | 内容 |
|---|---|
| 时间 | 2026-04-11 16:00 ~ 17:30（持续 90 分钟） |
| 影响服务 | gateway → backend-http → 对象存储 |
| 影响范围 | 固件包上传（>100MB 文件）全部失败，小文件上传正常 |
| 触发告警 | 无（没有对应的告警规则，靠用户反馈才发现） |
| 发现人 | 运营同事在群里问"是不是上传服务挂了" |

## 现象

16:00 开始，运维团队在后台批量上传新版本固件包（每个 ~300MB），所有上传都在 30 秒左右失败。日志显示 `object storage upload canceled: context canceled`，部分日志含 `io copy interrupted at offset=<N>`。

但**没有任何告警触发**——因为上传失败报的是 499（客户端关闭连接），不是 5xx，且 gateway 的 error rate 和 latency 指标都正常。

## 排查过程

| 时间 | 操作 | 发现 |
|---|---|---|
| 16:20 | 运营反馈后开始排查 | 查 backend-http 日志，看到大量 `context canceled` |
| 16:25 | 查 gateway 配置 | `gateway.upstream_timeout_ms = 30000`（30s） |
| 16:28 | 计算上传耗时 | 300MB / 实际带宽 ~10MB/s ≈ 30s → 刚好到网关超时 |
| 16:30 | 确认根因 | 网关超时（30s）< 大文件上传所需时间（~30s），网关在上传完成前掐断了连接 |

## 根因

**直接原因**：`gateway.upstream_timeout_ms=30000` 太小，无法覆盖大文件的上传时间。

**根本原因**：
1. 网关超时是全局统一的，没有按 path 区分——`/deploy`（需要 60s）和 `/upload`（需要 120s+）共用同一个超时值
2. 文件上传没有进度反馈，用户/运维不知道上传进行到哪了
3. 上传失败的告警维度只看 5xx，499 和上传耗时接近超时阈值的情况被忽略了

## 处置

1. **临时止血**（16:40）：把 `gateway.upstream_timeout_ms` 临时调到 120s，上传恢复。
2. **长期方案**：
   - 网关支持按 path 设置不同超时：普通接口、部署接口、上传接口分别配置
   - 大文件上传改为分片上传 + 断点续传
   - 补充 499 错误率的告警（虽然 499 是客户端断开，但量级突增说明服务侧有问题）
   - 上传接口改为异步模式：接收文件 → 给用户返回上传 ID → 后台慢慢传

## 教训

1. **"全局超时"是反模式**——不同接口的耗时预期完全不同，必须按接口类型分级。
2. **499（客户端关闭连接）不是只有"用户手贱关页面"**——服务端响应慢也是一种"被动 499"。不能完全忽略 499 的监控。
3. **上传功能必须给用户进度反馈**——否则用户不知道是"正在传"还是"卡死了"。
