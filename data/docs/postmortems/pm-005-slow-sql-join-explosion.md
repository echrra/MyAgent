# 故障复盘 PM-005：设备列表分页查询打爆数据库

## 基本信息

| 项目 | 内容 |
|---|---|
| 时间 | 2026-05-08 09:00 ~ 09:45（持续 45 分钟） |
| 影响服务 | backend-http → MySQL |
| 影响范围 | 设备列表页加载超时，设备管理后台无法使用 |
| 触发告警 | P0：慢查询速率远超基线 + 连接池接近耗尽 + 接口 P99 接近超时 |
| 发现人 | 监控告警 |

## 现象

早晨 9:00 上班后，设备列表查询接口 `/api/v1/devices/list` P99 延迟从 200ms 飙升到 30s。MySQL 的 `Threads_connected` 从 30 涨到 95（max=100），大量连接处于 "Sending data" 状态。

日志中大量 `query canceled: context canceled` + `driver: bad connection (ctx canceled)`。

## 排查过程

| 时间 | 操作 | 发现 |
|---|---|---|
| 09:05 | 告警触发 | 立即排查 MySQL processlist |
| 09:08 | SHOW FULL PROCESSLIST | 20+ 条相同的设备列表查询正在执行，每条扫描 rows > 500 万 |
| 09:10 | EXPLAIN 该查询 | `type=ALL`，没走索引——`key` 列为 NULL |
| 09:13 | 查慢查询日志 | 该 SQL 涉及 4 张表 JOIN + 子查询 + COUNT(*)，查询计划选择了全表扫描 |
| 09:15 | 确认根因 | 设备表数据量从 50 万增长到 200 万后，MySQL 优化器选择了不同的执行计划——放弃了原来的索引，走了全表扫描 |

## 根因

**直接原因**：设备表数据量突破 200 万后，MySQL 优化器自动选择了全表扫描（认为全表扫描比索引扫描快），但实际慢得多。

**根本原因**：
1. 设备列表查询 SQL 过于复杂（4 表 JOIN + 子查询），优化器决策空间太大
2. 没有在数据量增长后做 SQL Review——上次 Review 是 50 万行的时候，当时没问题就没关注
3. 没有对列表查询设置合理的 `LIMIT`——前端默认"展示全部"，但 SQL 没有强制分页

## 处置

1. **临时止血**（09:20）：
   - 强制使用索引 hint：`SELECT ... FROM devices FORCE INDEX(idx_status) WHERE ...`
   - 临时调大 `max_connections` 到 200
   - 09:30 查询恢复正常
2. **长期方案**：
   - 拆分复杂 SQL：去掉不必要的 JOIN，非必要字段改为异步补全
   - 强制分页：前端设 `pageSize` 上限 50，后端 SQL 强制 `LIMIT 50`
   - COUNT 改为异步统计（不阻塞列表查询）
   - 建立"数据量阈值 Review"机制：表数据量每增长一个量级或执行计划变化时，Review 关联 SQL

## 教训

1. **MySQL 优化器的"叛变"**——同样的 SQL、同样的索引，数据量变化后优化器可能选择完全不同的执行计划。没变代码不代表不会变慢。
2. **列表查询 + 4 表 JOIN + COUNT(*) 是定时炸弹**——这种 SQL 在数据量小的时候没事，一旦突破临界点就瞬间爆炸。
3. **分页必须有强制 LIMIT**——"让用户自己设 pageSize"等于没设。
