# 故障复盘 PM-007：并发设备导入导致 1062 唯一键冲突

## 基本信息

| 项目 | 内容 |
|---|---|
| 时间 | 2026-05-20 14:00 ~ 14:25（持续 25 分钟） |
| 影响服务 | admin → backend-http → MySQL |
| 影响范围 | 设备批量导入功能失败，个别设备注册失败 |
| 触发告警 | P1：admin_operation_duration_ms P99 突增 |
| 发现人 | 运营反馈"批量导入设备时好多条报错" |

## 现象

14:00，运营在后台批量导入 500 台新设备时，约 30 台导入失败。admin 日志出现大量 `Error 1062 (23000): Duplicate entry 'X000000000123' for key 'devices.idx_sn'`。

同时，backend-http 的设备注册接口也出现少量 1062 错误（约 5 条/min）——说明不是只导入功能有问题。

## 排查过程

| 时间 | 操作 | 发现 |
|---|---|---|
| 14:05 | 查 admin 导入日志 | 1062 错误集中在特定 SN 范围 |
| 14:08 | 查冲突的 SN | 这些 SN 在 13:58 左右已经被导入过一次——但那次导入"超时了"，运营点了重试 |
| 14:10 | 查重试逻辑 | admin 的导入接口：第一次请求 13:58 发起，DB 写入花了 8s（接近超时），前端 14:00 显示"超时"，运营点重试 → 第二次请求 14:00 发起，和第一次的写入并发 → 冲突 |
| 14:12 | 确认根因 | **前端超时=DB 没完成**。第一次请求在 DB 侧还在写入，前端已经提示"超时"了，运营的重试与第一次写入并发执行 → 1062 |

## 根因

**直接原因**：导入接口的超时时间（10s）太短，DB 写入需要 8-12s，前端超时后运营重试，两次写入并发导致唯一键冲突。

**根本原因**：
1. 前端超时和 DB 事务超时不是同一个值——前端 10s 就提示超时，但 DB 还在写
2. 导入没有幂等设计——同样的 SN 第二次导入应该跳过，而不是 INSERT
3. 导入接口没有"处理中"状态——运营不知道第一次请求还在处理，看到超时就点重试

## 处置

1. **临时止血**（14:15）：清理冲突数据，失败的 30 台设备手动一条条重新导入。
2. **长期方案**：
   - 导入 SQL 从 `INSERT` 改为 `INSERT ... ON DUPLICATE KEY UPDATE`（幂等）
   - 导入接口改为异步：接收文件 → 返回任务 ID → 后台处理 → 完成后通知
   - 前端显示"处理中"状态，禁止重复提交
   - 统一超时时间：接口超时 > DB 超时 > 业务最慢预期

## 教训

1. **1062 本质是幂等缺失的锅**——修复方式不是"让用户别点重试"，而是"点了重试也不会出错"。
2. **"前端超时 ≠ 操作失败"**——这是个经典的分布式系统认知。如果前端的超时时间小于后端的实际执行时间，"超时"只是前端放弃等待，操作可能还在进行中。
