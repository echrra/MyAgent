# 故障复盘 PM-011：定时任务调度冲突——两个 watcher 实例同时执行对账

## 基本信息

| 项目 | 内容 |
|---|---|
| 时间 | 2026-05-15 03:00 ~ 03:20（持续 20 分钟） |
| 影响服务 | backend-watcher（2 实例） |
| 影响范围 | 设备状态对账数据出现重复更新，部分设备状态被错误翻转 |
| 触发告警 | P1：watcher_event_lag_seconds 异常增长 |
| 发现人 | 监控告警触发后排查发现 |

## 现象

凌晨 3:00，`device_state_reconcile` 定时任务按计划触发。两个 watcher 实例同时执行了对账逻辑，导致：

1. 同一批设备的对账操作被执行了两次——第二次覆盖了第一次的结果
2. 部分设备因为并发更新出现状态不一致（watcher A 标记设备"离线"，watcher B 同时标记"在线"）
3. MySQL 出现少量 1062 和死锁（1213）

## 排查过程

| 时间 | 操作 | 发现 |
|---|---|---|
| 03:05 | 告警 | watcher_event_lag_seconds 从 2s 飙升到 45s |
| 03:08 | 查 watcher 日志 | 两个实例的日志里都有 `device_state_reconcile started`，时间戳相差不到 1s |
| 03:10 | 查 Redis 锁 | `lock:cron:device_state_reconcile` 存在，但值为空——锁已经过期了 |
| 03:12 | 分析锁竞争 | Redis 分布式锁的 TTL 是 60s，但对账任务实际执行需要 90s（设备量增长了）。锁在任务执行到 60s 时自动过期，实例 B 在 62s 时获取到了锁 → 两个实例并发执行 |
| 03:15 | 确认根因 | 分布式锁 TTL < 任务实际执行时间 → 锁提前过期 → 两个实例同时执行 |

## 根因

**直接原因**：Redis 分布式锁 TTL（60s）< 对账任务实际执行时间（90s），锁提前过期。

**根本原因**：
1. 锁的 TTL 是硬编码的，没有随设备量增长而调整
2. 锁没有"续期"机制（任务执行中定期延长锁的 TTL）
3. 任务执行时没有检查"锁是否仍然属于自己"

## 处置

1. **临时止血**（03:15）：手动终止实例 B 的对账任务，只留实例 A 执行。03:20 完成。
2. **数据修复**（03:20-04:00）：对比设备状态变更日志，手动修正被错误翻转的 12 台设备状态。
3. **长期方案**：
   - 锁增加续期机制：任务执行中按 TTL 的 1/3~1/2 周期续期，续期失败则主动停止任务
   - 任务开始时记录 start_time，如果发现锁不是自己持有的（被另一实例抢占），立即退出
   - 锁的 TTL 改为动态计算：取任务历史 P99 执行时间的 1.5 倍，并保留上限保护
   - 增加"定时任务锁竞争"的监控指标

## 教训

1. **分布式锁的"自动过期"是把双刃剑**——它防止了死锁（持有者挂了锁不释放），但也可能导致并发执行（持有者还活着但锁到期了）。
2. **锁 TTL 必须 > 任务最长可能执行时间**——不能只设一个"差不多"的值，且需要定期 Review 是否仍然合适。
3. **定时任务的锁必须有"续期"机制**——对于执行时间不确定的任务，续期是必不可少的。
